The need for this domain emerged from extensive user feedback through Builder satisfaction surveys, Developer console surveys, and user interviews about discoverability, search, support, documentation, and knowledge management.

DEEP DIVE

Redesigning Platform Navigation

The Sunrise platform had grown organically — new tools were added without a coherent navigation strategy, leading to a top navigation bar that couldn't scale. Builder surveys consistently ranked "discoverability" as the #1 pain point.

The Problem

With 20+ tools and growing, the horizontal navigation was overflowing. Users relied on bookmarks and Slack messages to find tools rather than the platform's own navigation. New team members reported a 2-3 week learning curve just to find the tools they needed.

The Approach

I conducted card sorting sessions with 12 builders across different roles, analyzed navigation analytics to understand actual usage patterns, and benchmarked internal developer platforms at other tech companies. We tested two concepts: a mega-menu approach and a persistent left-side navigation.

The Outcome

The left-side vertical navigation with grouped categories reduced tool discovery time by approximately 40%. Post-launch survey showed 72% of users found the new navigation "significantly better" than the previous version.

Scalable pattern: The navigation structure became the standard adopted across all new Sunrise sub-applications — one solution rolled out to the entire platform without redesign.

Key Design Decisions & Trade-offs

We chose vertical navigation over a mega-menu despite the mega-menu testing well initially.

The mega-menu scored higher in first-impression usability tests, but analytics showed that builders used the platform for extended sessions (avg. 45 min). A persistent left nav reduced context-switching cognitive load for power users. The trade-off was less screen real estate for content, mitigated by a collapsible sidebar.

We standardized creation flows even though teams had different requirements.

Each team wanted custom creation wizards for their specific resource types. I pushed for a unified creation pattern that balanced consistency with flexibility — a shared step structure (Name, Configure, Review) with pluggable configuration panels per resource type. This reduced the learning curve for builders working across multiple tools.

We integrated API Linter into the platform rather than keeping it as a standalone tool.

The API Linter team wanted to maintain their own UI. I argued that integration into Sunrise would dramatically increase adoption by meeting developers where they already worked. The trade-off was additional development effort for the integration, but adoption jumped from ~15 users (standalone) to 56 active users within weeks.

Key Impact

Improved developer experience for 1800+ active builders — 72% rated the redesigned navigation as "significantly better"

Unified fragmented tools and services into cohesive platform experience

Reduced tool discovery time by ~40% and increased API Linter adoption from 15 to 56 active users through platform integration

Reduced onboarding time for new team members from 2-3 weeks to under 1 week through improved navigation and documentation

Key Takeaway

As the sole designer for a platform serving 1,800+ builders, I learned that the most impactful internal tooling design isn't about individual features — it's about creating coherent systems that reduce cognitive load across the entire developer journey.

The navigation redesign alone had more impact than any single tool improvement because it changed how builders related to the entire platform. By establishing consistent patterns (unified creation flows, standardized navigation, integrated tools), I helped shift the Builder Platform from a collection of separate tools to a platform that teams actually wanted to use — measurably improving discoverability, adoption, and developer satisfaction.

Outcomes & Next Steps

What changed: teams moved from a tool-by-tool approach to a clearer platform experience with stronger discoverability.

What scaled: reusable patterns for navigation, creation, and ownership reduced repeated UX debt in subsequent initiatives.

Next: deepen in-product guidance and cross-service consistency so first-time builders complete critical workflows with less support dependency.
